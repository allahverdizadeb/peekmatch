# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

PeekMatch — an evidence-based job-application preparation platform for the global market, Azerbaijani-first (not a generic AI CV builder): landing → CV/vacancy analysis → free results dashboard (candidate-fit + CV-presentation-quality scores, real-vs-CV-visible match teaser, real premium-preview counts, one unlocked CV Change Plan example) → pricing (free tier + 2-package comparison table) → checkout (upgrade-diff pricing) → payment → paid Workspace (full report / CV Change Plan / Interview Playbook) → delete/expired lifecycle, plus legal pages and a shared-secret-gated admin panel (feedback viewing only — public submission was removed, see "Admin panel" below). Every recommendation is meant to trace back to the vacancy, the candidate's CV, or something the candidate explicitly confirmed — see "CV Change Plan and Truth Lock" below.

**Product-cleanup pass (post-launch simplification)** removed several UI surfaces that tested as clutter or redundant with other features, while explicitly preserving the backend/AI logic each one depended on where that logic is still load-bearing elsewhere: the free Results page's self-attestation Q&A block (purple "do you actually have this experience?" prompt), the "Müraciət səyahəti" (application journey) checklist on Results, the "Bu düzgün deyil" (flag as incorrect) action on CV Change Plan cards, the dedicated "Sübut Zənciri" (Evidence Chain) section on the Report tab, and the public "Təklif ver" feedback-submission widget. See the relevant sections below for exactly what was kept vs. removed for each.

**Cover Letter and CV Recheck were removed** (product update pass) — the product now ships exactly report / CV Change Plan / Interview Playbook. Their `Analysis.coverLetterJson`/`recheckCvText`/`recheckResultJson`/`recheckCount` DB columns are kept (commented `@deprecated` in `schema.prisma`) rather than dropped, to avoid a destructive migration; no application code reads or writes them anymore.

All user-facing copy and AI output is in Azerbaijani (error messages, prompts, schema enum values like `kritik`/`əsas`/`üstünlük`). Keep new user-facing strings in Azerbaijani unless working on the `outputLanguage` (az/en/ru) pipeline specifically.

There are two independent npm projects with no root package.json / workspace tooling: `backend/` and `frontend/`. Always `cd` into the relevant one before running npm commands.

The site UI is localized into Azerbaijani and English via `frontend/src/lib/i18n/` (`LanguageContext` + per-locale `Dict`-typed files under `locales/`, `LanguageSwitcher` in both headers). `az.ts` is the structural source of truth — `en.ts` is typed as `: Dict` so `tsc -b` fails the build if a key is added to one locale and not the other. The CV-analysis form's separate "AI result language" picker (`outputLanguage`, sent to the backend) tracks the current UI language live (`AnalysisForm.tsx`: `outputLanguage = outputLanguageOverride ?? uiLang`) until the user manually taps a pill, at which point it "pins" to their explicit choice and stops following further UI-language changes — this replaced an earlier version that only seeded once on mount and silently went stale if the user switched the site language before submitting. (Turkish and Russian were previously supported site-wide but were removed at the user's request — `outputLanguage`/UI language are now `'az' | 'en'` only. Any unrecognized/unsupported `outputLanguage` value — including legacy `'tr'`/`'ru'` rows — now falls back to **English**, not Azerbaijani: `PATCH /:id/settings`'s `resolveOutputLanguage()`, `docGen.ts`'s `resolveDocLang()`, and `ai.ts`'s `resolveLangName()` all default to `en`, reflecting the product's global-market positioning. This was flipped from the previous `az` default during the product-update pass.)

Since AI-generated content is baked into an `Analysis` row in whichever language it was created with — and the data model has no language-neutral structured layer to translate labels over, only narrative free text — switching the site's UI language after an analysis already exists does **not** retroactively translate it. `Results.tsx`/`Workspace.tsx` detect `info.outputLanguage !== uiLang` and render `components/LanguageMismatchNotice.tsx`, a banner explaining the mismatch and linking to `/analyze` to start a fresh analysis in the current language, rather than silently mixing languages or attempting an unrequested re-translation AI call.

| Variable | Purpose |
|---|---|

| `OPENAI_API_KEY` | Powers matching analysis, the CV Change Plan, Interview Playbook, and OCR (all via `lib/ai.ts`). Without it, every AI-backed function silently falls back to a deterministic offline placeholder (see below) so the rest of the flow stays testable. |

## Architecture

### Backend: stateful analysis pipeline, not CRUD

Everything hangs off one `Analysis` row (`backend/prisma/schema.prisma`). It's a single wide table used as a state machine, not normalized entities:

- `status`: `draft → processing → done/failed`, paired with `procStage` (0–6) that the frontend polls to drive a fake step-by-step progress UI (`Processing.tsx`).
- `vacancyStatus` is a **separate** state machine (`idle/loading/success/failed`) for vacancy URL extraction, independent of `status`.
- AI outputs are stored as opaque JSON strings (`resultJson`, `cvChangePlanJson`, `interviewPrepJson`) and parsed/typed on read using the `MatchResult`/`CvChangePlan`/`InterviewPrep` types from `src/lib/ai.ts`. Note `reportJson` exists in the schema but is currently unused — the `/report` endpoint derives its response directly from `resultJson`. `coverLetterJson`/`recheckCvText`/`recheckResultJson`/`recheckCount` are unused `@deprecated` columns left over from the removed Cover Letter and CV Recheck features — see above. `selfAttestedGapConfirmed`/`selfAttestedGapDetails` are a similar case as of the product-cleanup pass: the free Results page's self-attestation Q&A UI that used to *set* these fields (`PATCH /:id/self-attest`, called from `Results.tsx`) was removed, but the fields, the route, and the downstream AI-prompt threading (`buildSelfAttestedGap`/`selfAttestPromptNote` in `analyses.ts`/`ai.ts`) were deliberately left intact rather than deleted — the brief that removed the UI explicitly called for preserving internal logic a UI removal doesn't own outright. In practice this means the route is currently unreachable from the frontend and these two fields stay `null` for every new analysis; if a future UI re-wires the same endpoint, the CV Change Plan/Interview Playbook prompts will pick the answer back up automatically with no further backend change.
- Records self-expire: `expiresAt` = createdAt + 24h. `resolveAnalysis()` (in `lib/analysisLifecycle.ts`, shared by `analyses.ts` and `orders.ts`) returns a 3-way discriminated union (`{kind: 'ok'|'not_found'|'expired'|'deleted'}`) — `not_found` maps to 404, `expired`/`deleted` map to 410 with `{code}` so the frontend can render the right lifecycle state. `deletedAt` is a soft-delete tombstone set by `DELETE /:id` (user-triggered "delete my data") which also nulls every content field immediately but leaves `status` untouched (still `'done'`) — the row itself is still hard-purged later by the same 15-minute `setInterval` in `index.ts` once `expiresAt` passes, there's no separate retention policy for deleted rows. Any analysis-reading endpoint must go through `resolveAnalysis()`, not a raw `prisma.analysis.findUnique` — `orders.ts`'s `POST /` originally didn't, which meant a deleted analysis (content nulled, `status` still `'done'`) could still pass its `status !== 'done'` check and let a user "buy" already-wiped content; fixed by routing it through the shared resolver too. On the frontend, `err.status === 410` (with `err.code`) renders the shared `LifecycleState` component (`Results.tsx`, `Workspace.tsx`, `Checkout.tsx`); a plain 404 gets a simpler inline "not found, start a new analysis" state instead — `Pricing.tsx` is the one page that still silently swallows analysis-fetch errors (`.catch(() => {})`) since it can render its package grid without the analysis loaded.
- `runAnalysis()` in `analyses.ts` is fire-and-forget: `POST /:id/start` responds immediately, then a background async function sleeps through fake `procStage` increments before calling the real AI. The frontend polls `/status` and `/result` — there's no websocket/SSE.
- The `Order` model is separate and cumulative-tiered, now **2 tiers** (was 3 — collapsed in the product-repositioning pass, see below): package 2 implies ownership of 1 (`highestOwnedPackage` in `lib/pricing.ts`, which also defensively caps any stale `package: 3` row from before the restructure down to 2). Every paid endpoint re-derives "what's owned" from `Order` rows with `status: 'paid'` (via the shared `ownedPackages()` helper, also in `lib/pricing.ts`) — ownership is never cached on `Analysis`. Buying a higher tier while already owning a lower one charges only the difference (`upgradePriceUsd()`, computed server-side in integer cents in `POST /orders` — never trust a client-supplied amount). Gating helpers are `unlocksApplication(owned)` (report, CV Change Plan, Evidence Chain — package 1+) and `unlocksInterview(owned)` (Interview Playbook — package 2 only); there's no `unlocksReport`/`unlocksCv` split anymore.
- Paid content (Interview Playbook) is generated as a real, DB-backed fire-and-forget job — `Analysis.interviewPrepStatus` (`idle | processing | done | failed`) + `interviewPrepFailReason`, mirroring `runAnalysis()`'s own pattern rather than one long-lived blocking request. `POST /:id/interview/generate` claims the job atomically (a single WHERE-guarded `updateMany` on `interviewPrepStatus IN ('idle','failed')` — only the caller that actually flips the row starts real work, so this is safe even under true concurrency and survives a server restart, unlike the in-memory de-dup `Map` this replaced), `GET /:id/interview/status` is a lightweight poll target, and `GET /:id/interview` returns the finished result only (409 until `done`) — an already-completed Playbook still returns immediately with no extra work. The **CV Change Plan is the one exception** — it's generated *eagerly*, inside `runAnalysis()` right after `analyzeMatch`, for every analysis regardless of payment, because the free result needs real (non-hardcoded) change/risk counts; `ensureCvChangePlan()` still exists as a lazy fallback for when the eager call fails or after a self-attestation change invalidates the cache. `ensureCvChangePlan`/`generateInterviewPrep` thread `Analysis.selfAttestedGapConfirmed` + `selfAttestedGapDetails` into the AI prompt when present (see the note on these fields above — currently always `null` since the capturing UI was removed) — changing that answer after generation nulls the cached JSON columns so the next fetch regenerates with the new answer.

### Payments are structurally real, financially fake

`orders.ts` implements real order records, gating, and a real pending→processing→paid/failed transition — but `POST /orders/:id/simulate` just waits 1.4s and flips status based on a client-supplied `outcome` field. There is no live payment provider. Don't assume a webhook or provider SDK exists anywhere.

### AI integration (`backend/src/lib/ai.ts`)

**Provider: OpenAI, via the Responses API — not Anthropic.** This was `lib/anthropic.ts` using `@anthropic-ai/sdk` until a provider migration; the file was renamed (not just edited) to `ai.ts` to stay provider-neutral. All structured generation functions (`analyzeMatch`, `generateCvChangePlan`, `generateInterviewPrep`, plus the vision-based `ocrDocumentImages`) go through one shared helper, `createStructured()`, which calls `client.responses.create({ model, reasoning: { effort: 'medium' }, store: false, input: [...], text: { format: { type: 'json_schema', name, schema } } })` and validates the result before returning it. (`generateCoverLetter` used to be a fifth function on this list — removed along with the Cover Letter feature.) When adding a new AI-generated artifact, follow this pattern (add a schema + call `createStructured`) rather than free-text parsing.

**One model, not two.** `MODEL` reads `process.env.OPENAI_MODEL`, falling back to `gpt-5.6-terra`, and is used for every call — there is no cheap/expensive split anymore. (The old Claude-era split — a cheap `MODEL` for flatter schemas and an expensive `ANALYSIS_MODEL` for schemas with large nested-object arrays — existed because of a specific, verified Claude Haiku bug where nested free-text fields ignored `outputLanguage`. That bug was never verified against GPT-5.6, and per the migration's explicit brief only `OPENAI_API_KEY`/`OPENAI_MODEL` are read from config, so this codebase does not re-introduce a tier split. If a similar nested-field language bug ever turns up on `gpt-5.6-terra`, the fix is the same class of investigation: a real non-Azerbaijani end-to-end run inspecting `requirements[].title`/`.evidence` in the actual response, not a same-language test.)

**Output length is bounded by the prompt, not `max_output_tokens`.** Verified against OpenAI's own docs during the migration: reasoning models on the Responses API (this family included) reject `max_output_tokens` outright rather than just capping output, so none of the calls in this file set it. Every schema instead has explicit per-section/per-array count limits spelled out in the user prompt (e.g. `generateInterviewPrep`'s "3-5 HR sualı, 3-5 situasiya sualı, 3-5 texniki sual — CƏMİ 10-15, hər kateqoriyada ən çoxu 5") — this is a carryover of a real lesson from the Claude era, where an ambiguously-worded "10-15 questions" prompt (no per-category cap) let the model generate a response so large a real call hung for **~58 minutes** before the SDK's connection timeout fired. If a schema grows (more fields per array item, more items requested), re-verify with a real end-to-end call and watch wall-clock time, not just JSON validity.

**`generateInterviewPrep` is split into FOUR parallel calls, one per question category plus one "support" call.** It went through two prior shapes before this one: originally one large sequential call (~3 minutes), then two parallel calls — `interviewCoreSchema` (all three question categories combined) + `interviewSupportSchema` — which still routinely took 60-95s and could outright fail (a real live run hit `interview_prep_core`'s 75s timeout and failed). Live, controlled benchmarking (see `INTERVIEW_PLAYBOOK_FIX_REPORT.md`) isolated *output size* (item count × fields per item) as the dominant latency lever for this model — reasoning effort and prompt size alone weren't enough on their own. It's now `interview_hr` (`strongestTopic`/`biggestRisk`/`tellMeAboutYourself`/`hrQuestions`), `interview_situational`, `interview_technical`, and `interview_prep_support` (`criticalGapStrategies`/`starStories`/`cvVerificationQuestions`/`gapExplanations`/`questionsToAsk`, unchanged shape) — 4 calls issued concurrently via `Promise.all`, merged into one `InterviewPrep` object with the exact same shape callers always got. The question item schema also dropped `importantPoints`/`missingInformation`/`commonMistakes` (not part of the product's required content), and the prompt sends a condensed context object (filtered requirements + capped strengths, no raw vacancy text) instead of the complete raw CV/vacancy text. Each call passes `timeoutMs: INTERVIEW_CALL_TIMEOUT_MS` (28s) and `reasoningEffort: 'low'`. Measured live end-to-end after this change: **~19s backend / ~21s including frontend polling** for a real analysis — reliably under both the 20s target and 30s acceptable max, down from routine 60-95s+ (and outright failures) before.

**`timeoutMs` also forces `maxRetries: 0` on that call — this is load-bearing, not cosmetic.** Verified live: the OpenAI SDK defaults to `maxRetries: 2`, and its own retry logic treats a connection *timeout* as retryable. Left at the default, a `timeoutMs` value doesn't cap wall-clock time at all — each retry gets its own fresh budget, so a real call was observed taking several minutes to finally surface `AiError('timeout', ...)`, silently defeating the documented timeout budget. `createStructured()` now passes `maxRetries: 0` alongside `timeout` whenever `timeoutMs` is set, so one attempt is the real, whole story. If a future caller passes `timeoutMs` and actually wants retries, this needs to become its own explicit option rather than an implicit side effect.

**The Interview Playbook job is now a real, DB-backed status machine, not one long-lived request.** See the `Analysis.interviewPrepStatus` note above (`POST /:id/interview/generate` / `GET /:id/interview/status` / `GET /:id/interview`). `Workspace.tsx`'s `InterviewPlaybookTab` starts the job (idempotent — safe on every mount/refresh) and polls status every 1.8s; there is **no elapsed-time tracking, no seconds, no countdown, and no estimated percentage anywhere in this flow** (a product requirement, not an oversight — an earlier version had a client-side estimated-progress/reassurance model keyed off elapsed seconds; it was removed entirely because even *estimated* time-based copy was judged the wrong signal). The stage label and progress-bar position instead cycle on a fixed decorative interval that carries no timing information at all — purely so the wait doesn't feel frozen, same spirit as `RotatingHint` (used identically by `CvPlanTab` on this same page). A poll-attempt cap (not time-based, never surfaced to the user) guards against a genuinely stuck job (e.g. a server restart mid-generation). On a real `failed` status, the UI shows one calm, fixed sentence with **no retry button and no user action requested** — nothing in the component auto-restarts generation; a page refresh or tab-switch remount simply re-calls the idempotent start endpoint, which is a no-op if a job is already `processing` or `done`.

**Every structured response is validated before a caller sees it** (`parseStructuredResponse()` in `ai.ts`) — never a bare `JSON.parse`. Checks, in order: an `incomplete` response status (truncated output, or a `content_filter` refusal), an explicit `refusal` content block, empty output text, invalid JSON, and missing required top-level keys (reusing each schema's own `required[]` array, so the two can't drift apart). `analyzeMatch` additionally clamps `compatibility`/`cvPresentationScore`/`hrScreeningEstimate`/`realCompatibility` to `[0, 100]` after parsing — `compatibility` gets fully overridden anyway by `scoring.ts`'s deterministic computation, but `cvPresentationScore` has no such override (it's an intentionally independent axis), so it's the one score that genuinely needs the defensive clamp.

**Errors are typed, not raw.** Every OpenAI SDK exception (auth, rate limit vs. insufficient-quota, timeout, network, unavailable model/permission-denied) and every validation failure above gets normalized into an `AiError` with a `code` (`auth_error`/`insufficient_credits`/`rate_limited`/`timeout`/`network_error`/`model_unavailable`/`refusal`/`invalid_response`/`unknown`) and an already-safe, Azerbaijani, user-facing `.message` — no request/response internals or the key ever reach a caller. Route handlers in `analyses.ts` that trigger generation catch this via a shared `respondAiError()` helper, which maps `.code` to an HTTP status (503 for transient/retryable, 502 for an upstream response problem, 500 for operator-side config/billing) and returns `{ error, code }`. `runAnalysis()`'s background job (no HTTP response to send) instead writes `AiError.message` straight into `Analysis.failReason`, which the frontend already polls and displays.

Every function has a matching offline fallback (`offlineAnalyze`, or inline stub objects) used when `getClient()` returns null (no API key). These fallbacks must stay wired up — they're what makes the whole flow demoable without a key, and they're clearly labeled as placeholders in the returned copy (e.g. `criticalGapSummary: 'AI konfiqurasiya olunmayıb...'`).

Free vs. paid content is a **field-level split on the same `MatchResult`**, not separate AI calls: `GET /:id/result` returns a curated subset of fields (`compatibility`, the independent `cvPresentationScore`/`cvPresentationLabel` axis, `realCompatibility` number, category scores, strengths, a 2-item requirements preview, plus app-computed `cvChangesSummary`/`interviewRisksCount`/one unlocked `exampleCard` — see below); `GET /:id/report` returns the full `requirements` array plus `weakPresentation`/`improvementOpportunities`/`realCompatibilityGap` (the *explanation* of the real-vs-visible gap — the number is a free teaser, the why is report-gated), gated by `unlocksApplication(owned)`. When changing `matchResultSchema`, check both response shapers in `analyses.ts` for what needs to move across the free/paid line. `realCompatibility` is defensively clamped to `>= compatibility` right after parsing the AI response, since nothing in JSON-schema output enforces that inequality; `cvPresentationScore` has no such cross-field constraint — it's an intentionally independent axis (a well-qualified candidate can still score low here if the CV undersells them).

### CV Change Plan and Truth Lock

The old "Tailored CV" (which silently generated a whole replacement CV) was replaced with the **CV Change Plan**: `generateCvChangePlan()` returns a list of evidence-cited change cards (`CvChangeCard`: `section`, `currentText`, `whatToChange`, `problem`, `recommendedText`, `relatedRequirements[]`, `evidenceFromCv[]`, `priority` (reuses the existing `kritik`/`əsas`/`üstünlük` enum), `changeType` (`rewrite`/`add`/`clarify`/`remove`)) instead of a full document. **Truth Lock** — the product rule that no recommendation may be invented — is enforced in *application code*, not just prompted: `scoring.ts`'s `sanitizeCvChangePlan()` drops any `rewrite`/`add`/`remove` card whose `evidenceFromCv` is empty (a concrete claim with zero cited evidence is exactly the fabrication risk this exists to prevent); `clarify` cards are exempt since citing no evidence is the point of asking the candidate to confirm something.

**Cards are concise by design, not just by prompt.** `whatToChange` (one short imperative sentence — "what to do") and `problem` (one short sentence — "why it matters") are each length-capped in `sanitizeCvChangePlan()` (character-truncated with an ellipsis, same function as the evidence check above), and `recommendedText` is word-capped (~100 words) — a model that ignores the prompt's own "max 1 sentence" / "60-90 words" guidance gets safely shortened rather than shown unbounded. `frontend/src/components/CvChangeCard.tsx` renders one card (shared between the free Results page's single unlocked example and the paid Workspace `cv-plan` tab's full list): `whatToChange`/`problem`/the recommended-text block are shown by default, while `currentText`/`relatedRequirements`/`evidenceFromCv` sit behind a single "Ətraflı məlumat" (more details) toggle — progressive disclosure so a page of 6-10 cards stays scannable. Per-card completed/edited state is `localStorage`-only (`lib/localCardState.ts`) since there's no backend entity per interaction; there is no longer a "flag as incorrect" action (removed — see the product-cleanup note near the top of this file) — `CardState` only tracks `completed`/`editedText` now.

**Evidence Chain** (`backend/src/lib/evidenceChain.ts`, `GET /:id/evidence-chain`) is a pure derived view, not a new AI call: it cross-references requirement titles already present across `resultJson`/`cvChangePlanJson`/`interviewPrepJson` (fuzzy-matched — case-insensitive substring, since the AI doesn't reproduce byte-identical titles across separate calls) to link requirement → evidence → status → related CV change → related interview question. The dedicated "Sübut Zənciri" section that used to render this on the Report tab was removed in the product-cleanup pass, but the backend route and `lib/evidenceChain.ts` were deliberately kept — `components/RequirementPriorityMap.tsx` (the Report tab's main requirement view) still fetches `GET /:id/evidence-chain` and uses `relatedChangeSection` to power its "Növbəti addım" (next step) pointer for urgent requirements ("CV Dəyişiklik Planında '…' bölməsinə baxın"). Don't remove this endpoint without first checking `RequirementPriorityMap.tsx` — it's a second, still-active consumer, not dead code.

**Cover Letter and CV Recheck have been removed** from the product (see the note near the top of this file). `scoring.ts`'s fuzzy-title-matching helper that CV Recheck's diff used to reuse (`titleMatches`/`wordOverlapRatio`) was removed along with it — Evidence Chain has its own simpler (exact/substring, not fuzzy-word-overlap) `titleMatches()` local to `evidenceChain.ts`, unaffected by that removal.

### Admin panel (`backend/src/routes/suggestions.ts`, `middleware/adminAuth.ts`)

Standalone `Suggestion` model (no relation to `Analysis`/`Order`, no expiry; `category`/`text`/`email` columns) read by an `ADMIN_KEY`-gated `GET /api/suggestions`. The public submission path — `POST /api/suggestions`, the frontend's floating "Təklif ver" widget (`components/FeedbackWidget.tsx`), and the `lib/mailer.ts` email-notification side-effect — was removed in the product-cleanup pass; the route file, component, and `mailer.ts` no longer exist. `GET /api/suggestions` was kept rather than removed alongside it, because it's a second, still-active dependency: `AdminGate.tsx` validates the `ADMIN_KEY` by calling it, and `Admin.tsx` still displays whatever rows already exist. The `Suggestion` table/migration are untouched (no new rows will ever be created now, but nothing reads that as an error) — a future controlled migration can drop it once the admin panel itself is retired. The admin frontend's "AI structuring" (theme/sentiment/priority tagging) is deliberately fake — deterministic keyword matching (`frontend/src/lib/suggestionClassifier.ts`), not a real AI call — matching the original design; don't assume it's an LLM classifier.

### Document generation (`backend/src/lib/docGen.ts`)

PDF only (the report), via Playwright/Chromium rendering hand-built HTML strings. The Chromium executable path is overridable via `PLAYWRIGHT_CHROMIUM_PATH` (for container/deploy images with a fixed browser location); unset, it falls back to Playwright's own auto-detected install (`chromium.executablePath()`), which is what local development uses. DOCX generation (via the `docx` library) existed only for the Cover Letter download and was removed with that feature — `docx` is no longer a dependency.

### CV/vacancy extraction

- `lib/cvParse.ts`: PDF via `pdfjs-dist` (direct dependency — do not reintroduce `pdf-parse`; it bundles its own older nested `pdfjs-dist` whose worker collides with ours the moment both run in the same process, throwing "API version does not match Worker version"), DOCX via `mammoth`. When a PDF's extracted text falls below `MIN_EXTRACTED_TEXT_CHARS`, it's not immediately rejected — each page is rasterized via `pdfjs-dist` + `@napi-rs/canvas` (at viewport scale 2 — benchmarked lower scales against a real image-only resume and confirmed they degrade OCR accuracy, e.g. dropped/garbled words, so don't lower this to chase speed) and sent through `ai.ts`'s `ocrDocumentImages()` (vision OCR via the Responses API's `input_image` content type, same `OPENAI_API_KEY`/offline-fallback convention as the rest of the AI pipeline) before falling back to the `scanned_pdf` rejection. This is what makes image-only PDFs — resumes exported as a rasterized page by design tools (Canva, Figma, browser print-to-PDF of a styled template) rather than real text — work despite having zero extractable text layer. Only then does it reject non-PDF/DOCX and genuinely-empty documents (below `MIN_EXTRACTED_TEXT_CHARS` even after OCR). This is a blocking ~10s+ request on `POST /api/analyses` for image-only PDFs specifically (normal digital-text PDFs are near-instant, no OCR involved) — if upload latency becomes a complaint again, the next lever is making the upload itself async/pollable (matching the `runAnalysis()` fire-and-forget pattern).
- `lib/vacancyExtract.ts`: renders the vacancy URL with a real headless browser (`playwright`'s `chromium.launch()`, same `PLAYWRIGHT_CHROMIUM_PATH`-overridable pattern as `docGen.ts`), then `cheerio` scrapes the rendered DOM. This is not a raw `fetch()` — it was originally, but that only ever saw the pre-hydration HTML shell on JS-rendered (SPA) job boards, silently returning site-navigation/category-listing noise instead of the actual posting (confirmed live against jobsearch.az: the AI itself correctly flagged the input as "not a specific vacancy," which is what led to finding this). A headless browser executes the page's JS like a real user's browser would, so this works for both server-rendered and client-rendered sites. UA spoofing carries over; `page.waitForTimeout(1500)` after `domcontentloaded` gives SPA hydration a moment before scraping. SSRF protection (see `ssrfGuard.ts`) is enforced via `page.route()` intercepting every navigation request (initial URL and every redirect hop) and resolving+checking its host before Playwright is allowed to connect — this is stricter than a plain `fetch()` could give you, not weaker, despite the browser-based rewrite. Because rendering noisy pages can produce a much longer vacancy text than a plain fetch would (nav chrome, "similar postings" widgets, etc., all get scraped alongside the real content — there's no site-specific content-isolation, by design, to stay general), `analyzeMatch()`'s `max_tokens` was raised from 8000 to 16000 after a real noisy page pushed a response past the old budget and truncated the JSON mid-string; if that happens again on some other page, the fix is tighter vacancy-text extraction, not another `max_tokens` bump. Falls back to manual paste (`POST /:id/vacancy/manual`, min 3000 chars) when a site blocks scraping or returns too little text — this fallback path is a first-class part of the product, not an error state to eliminate.

### Prisma setup specifics

Uses Prisma 7's `prisma-client` generator (not the classic `prisma-client-js`) with a custom `output` into `backend/src/generated/prisma/` — that directory is generated code, checked into `src/` but not meant for manual edits. The client is instantiated via `@prisma/adapter-better-sqlite3` in `db.ts` rather than the default driver. `prisma.config.ts` (not `schema.prisma`'s `datasource` block) is where the migrations path and datasource URL are configured — check there first if migrations behave unexpectedly.

### Frontend: linear wizard over React Router

Routes in `App.tsx` mirror the product funnel in order: `/` (Landing) → `/analyze` (AnalysisForm: CV upload/paste + vacancy URL/paste + language/consent) → `/processing/:id` (polls status) → `/results/:id` (free dashboard) → `/pricing/:id` → `/checkout/:id/:pkg` → `/payment/:orderId` → `/workspace/:id/:tab` (paid deliverables: report/cv-plan/interview, tab-routed), plus standalone `/privacy` `/terms` `/deletion` (legal, lazy-loaded) and `/admin` `/admin/insights` (key-gated, see below).

Analysis lifecycle 410s (`expired`/`deleted`, see backend section) surface inline via a shared `LifecycleState` component wherever `api.ts`'s `req()` wrapper detects `err.status === 410` — there are no dedicated `/expired` or `/deleted` routes; "delete my data" is a confirm modal on Results/Workspace, not a separate page. This is a deliberate simplification over the original design's per-state screens.

Admin pages (`Admin.tsx`, `AdminInsights.tsx`) are wrapped in `AdminGate.tsx`, which prompts for the `ADMIN_KEY` client-side, validates it by calling `GET /api/suggestions`, and caches it in `sessionStorage` (not `localStorage`) so it doesn't linger across browser sessions. `lib/adminApi.ts` attaches the stored key to admin requests automatically.

`frontend/src/lib/api.ts` is the single fetch client — every backend response shape has a matching TypeScript interface here that mirrors the backend's hand-shaped JSON (not auto-generated from Prisma or an OpenAPI spec). When a backend response shape changes, update the corresponding interface here by hand.

Styling is Tailwind v4 via `@tailwindcss/vite` (no `tailwind.config.js` — v4 uses CSS-based config, check `index.css` for `@theme`/token definitions) mapped to the original design's color/spacing/radius/shadow tokens. Linting is `oxlint`, not ESLint — don't add `.eslintrc`.
